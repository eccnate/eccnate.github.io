Why We Love C

C is lingua franca. The Godfather. There are no "there's no library for that yet" excuses. You will find that most language runtimes are merely built atop C, waiting for capabilities to be written in C. What-a-shit-show. (Pity the fool who says it cannot be done in C!) C is the go-to foundation for real engineers developing new technology, making it ubiquitous within the tools one uses today. How the C runtime implements ideas is how POSIX systems as-a-whole get implemented (Linux, Android, Chrome Books, Raspberry Pi, Internet servers, many others!) The two specifications were drafted simultaneously and have been in bed together ever since. Competing languages will try to compare, but C is the veteran in the room that has done it all. Micro-managing code to optimize algorithms makes C binaries the fastest practical processes [benchmarks]. Being just a step above Assembly, it is expressive as a high level language, while having the ability to cater to the hardware. The real power goes to those with the faculties to master its domain, as one is made responsible for the binary they produce. The training wheels are off and there is no safety net! (Not true anymore, the OS has many built-in fail-safes.) There are no cutting corners, it demands thoroughness. Use it wisely. The first principle in the "spirit of C" is to trust the programmer.

The Down Low

What distinguishes C?

  • Manual memory management
  • No garbage collection
  • Minimal tools built-in; streamlined
  • Not portable (despite early claims compared to Assembly upon inception)
  • Is in fact a high level language
  • Often is the runtime system

When and how to C

"C should not be your first rodeo."
Motherduck

The distinguishing factors of C make it appropriate for projects with low level considerations of systems, and reduntant otherwise. If one is not going to enjoy venturing into the abyss of systems, there are perfectly suited languages for that purpose of which to spend one's time. Do not let tutorials cop-out and convince oneself that C is just syntax like most other languages. Writing robust battle-ready code for production and showcasing is going to require a broader understanding of the toolchain than just that of the C language itself. To require a sub-system like Cygwin/MinGW is the antithesis of a C programmer. This is where the programmers are distinguished from the script-kiddies.

Reading a C-standard textbook from cover to index will fill one's head with un-applicable knowledge. Unbuffered keyboard input from the user is not acquired through the C language alone, never mind audio/visual and device communications. (And static programs won't get one laid.) Once one breaks those barriers the code will –initially– only be valid within the environment of which it was produced. Fuck.

Start with the preprocessor and the linker—learn the toolchain! It's unsexy but there's no point in learning C if one is not going to master its domain. To master a domain, one must work from the ground–up. Learn the preprocessor and the linker, be a programmer!
